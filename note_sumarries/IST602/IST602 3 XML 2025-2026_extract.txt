 3.1 Introduction/Basics
• Widely supported open (i.e., non-proprietary) technology for describing data that has 
become the standard format for data exchanged between applications over the Internet
• Permits document authors to create markup for virtually any type of information
• Can create entirely new markup languages that describe specific types of data, including 
mathematical formulas, chemical molecular structures, music and recipes
• Describes data in a way that human beings can understand and computers can process

 3.2.2 XML Vocabularies (i.e., XML-based markup languages)
• Provide a means for describing particular types of data in standardized, structured ways. 
• Examples: XHTML (Extensible HyperText Markup Language), MathML™ (for 
mathematics), V oiceXML™ (for speech), CML (Chemical Markup Language – for 
chemistry), XBRL (Extensible Business Reporting Language – for financial data exchange), 
others
• Much of the massive amounts of data that are currently stored on the Internet in various 
formats (databases, web pages, text files, etc) are transformed into XML for the purpose of 
transferring them between systems
• Industry groups are developing custom XML vocabularies for most major industries that 
will allow business applications to communicate in common languages, e.g., using web 
servicess
 3.2.3 Viewing and Modifying XML Documents
• An XML document is a text file that usually ends with the .xml filename extension
• Does not require special software to view
• Any text editor that supports ASCII/Unicode characters can open XML documents for 
viewing and editing

 3.2.4 Processing XML Documents
• Requires software called an XML parser (or XML processor)
• A parser makes the document’s data available to applications
• XML parser: Identifies components of XML documents and stores them in a data structure 
for manipulation
• While reading an XML document’s contents, a parser checks that the document follows the 
syntax rules specified by the W3C’s XML Recommendation (www.w3.org/XML)
◦ Case sensitivity
◦ Single root element, a start tag and end tag for each element, and properly nested tags
• Well-formed XML document
◦ A document that conforms to the syntax rules
◦ Parsers can provide access to XML-encoded data in well-formed documents only

• An XML document with correct syntax is called "Well Formed".
◦ XML documents must have a root element
◦ XML elements must have a closing tag
◦ XML tags are case sensitive
◦ XML elements must be properly nested
◦ XML attribute values must be quoted

• Well-formed XML documents are not necessarily valid

◦ Well formed and
◦ Must conform to a document type definition

◦ XML Schema - An XML-based alternative to DTD

(i.e., its elements contain the proper attributes and appear in the proper sequence)

Character Data vs. Parsed Character Data
• Keyword CDATA (e.g., Line 9)
◦ Specifies that the attribute (type in this example) contains character data (i.e., a string)
◦ A parser will pass such data to an application without modification
• Keyword #PCDATA (e.g., Line 11)
◦ Specifies that an element (name in this example) may contain parsed character data
▪ i.e., data that’s processed by an XML parser
◦ Elements with parsed character data cannot contain markup characters, such as <, > or &
◦ Any markup character in a #PCDATA must be replaced with the character’s 
corresponding character entity reference
◦ Examples
▪ Character entity reference &lt; should be used in place of the less-than symbol (<)
▪ Character entity reference &gt; should be used in place of the greater-than symbol 
(>)
◦ Note
▪ To use a literal ampersand, the corresponding entity reference &amp; should be 
used instead
• Parsed character data can contain ampersands (&) only for inserting entities
